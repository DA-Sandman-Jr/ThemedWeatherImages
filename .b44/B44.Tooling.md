# B44 Tooling Traps

This file is generated from canonical `B44.Standards` guidance. Change the source in B44.Common rather than editing a consumer's generated copy.

## Formatting and analyzers

- A `dotnet format --verify-no-changes` validation run is different from an automated fixer run. For fixes, use `dotnet format whitespace <solution>` plus narrowly targeted `dotnet format style <solution> --severity info --diagnostics <known-fixable IDs>` invocations.
- Do not put `IDE0130` or `MA0051` in a fixer diagnostic list. Their current fix paths are unsupported; repair namespace/folder mismatches and overlong methods deliberately by hand.
- Analyzer severities belong only in B44.Standards' packaged globalconfig. A repository `.editorconfig` outranks global configs and can create an unresolvable policy conflict.

## Build isolation

- Concurrent repositories can cross-talk through reused MSBuild nodes and the shared compiler server on this machine. Run build and test verification with `/nodeReuse:false -p:UseSharedCompilation=false -m:1` so another repository's diagnostics cannot contaminate the result.

## Baselines, text, and packages

- Before measuring a clean baseline, remove only the intended repository's resolved `bin` and `obj` directories. Build products survive checkout and reset and can fake the result.
- Preserve each edited file's existing encoding, BOM, final newline, and line-ending style. Do not let a bulk text-mode rewrite silently normalize unrelated lines.
- Immediately after publishing a new package version, use `dotnet restore --no-cache` in its first consumer validation so a stale local HTTP index cannot masquerade as the new package being absent.
